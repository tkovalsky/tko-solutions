# Roadmap: Human API Risk Assessment

> **Status — supporting reference, reframed 2026-06-25.** "Human API" is a **finding type and
> diagnostic lens**, not a standalone offer, separate assessment, or pricing tier. The offers are
> the **Operational Recovery Assessment** and the **Prior Authorization Operational Assessment**
> (see the authoritative [`/CURRENT_REALITY.md`](../CURRENT_REALITY.md)). Read the sequencing below
> as roadmap input for those assessments, not for a separate Human API product.

## Roadmap Principle

This roadmap separates current reality, hypothesis, validation work, and possible future software. The immediate objective is a coherent advisory offering, not application development.

## Current State

### What Exists Today

- TKO positioning around operational truth and operational intelligence.
- Healthcare domain credibility in payer operations, prior authorization, utilization management, care management, interoperability, and transformation governance.
- RachelOS as a proof pattern for memory, facts, state, canonical action, and human approval.
- Draft offer architecture for diagnostics, decision-layer work, and fractional advisory.
- Existing repository documentation that supports the broader TKO thesis.
- This baseline documentation set for Human API Risk Assessment.

### What Is Hypothesis

- Healthcare payer operations buyers will recognize Human API dependency as an urgent risk category.
- Human API Risk Assessment can be sold as a standalone diagnostic.
- The scoring model will create enough clarity to support executive decisions.
- Assessment findings will convert into decision-layer and advisory engagements.
- Human API language will resonate without sounding dehumanizing if framed carefully.

### What Requires Validation

- Buyer willingness to pay at the proposed price ranges.
- Which buyer owns the budget: operations, transformation, IT, AI, or consulting partner.
- Which trigger event creates highest urgency.
- Whether assessment scope should be workflow-specific or function-wide.
- Whether the scoring model is understandable and credible in executive readouts.
- Which case study patterns are publishable.

### What Could Eventually Become Software

Only after advisory validation:

- Assessment intake tool
- Interview evidence repository
- Dependency mapping workspace
- Risk scoring dashboard
- Workflow dependency graph
- Recommendation library
- Action plan tracker
- Progress reassessment tool
- Benchmarking dataset

No software should be built before the assessment method is validated manually.

## 90-Day Plan

### Days 1-30: Package The Offer

- Finalize assessment narrative and one-page offer description.
- Convert this documentation into an internal sales brief.
- Create executive readout outline.
- Create interview workbook from the question library.
- Create scoring worksheet using the assessment framework.
- Draft first three thought leadership essays.
- Identify 20 to 30 target accounts or partner conversations.

Validation focus:

- Does the Human API category resonate?
- Which pain language gets the strongest response?
- Which buyer titles lean in?

### Days 31-60: Run Market Conversations

- Conduct buyer discovery calls with healthcare operations, transformation, health tech, and consulting leaders.
- Test offer names and pricing ranges.
- Test whether buyers prefer Human API Recovery Assessment or Operational Truth Diagnostic as the entry point.
- Refine scoring dimensions based on buyer feedback.
- Build anonymized case examples from approved evidence.
- Publish initial thought leadership.

Validation focus:

- What budget source exists?
- What trigger event creates urgency?
- What evidence is required to buy?

### Days 61-90: Pilot The Assessment

- Secure one paid or tightly scoped pilot assessment.
- Use manual documentation, interviews, artifacts, and scoring worksheets.
- Deliver executive readout and 90-day action plan.
- Capture lessons learned.
- Revise assessment framework and deliverables.
- Identify follow-on decision-layer or advisory scope.

Validation focus:

- Can the assessment produce executive-grade clarity?
- Are findings actionable?
- Does remediation convert to follow-on work?

## 6-Month Plan

### Offering Maturity

- Complete 2 to 3 assessments.
- Standardize deliverable templates.
- Refine risk scoring language and weighting.
- Package Prior Authorization Operational Assessment as a Tier-1 healthcare wedge, focused on administrative burden, denial reduction, workflow quality, operational consistency, Human API risk, and Gold Card readiness as an output.
- Build a small anonymized finding and recommendation library.
- Create a partner version for consulting firms.
- Publish a flagship Human API essay or report.

### GTM Maturity

- Define primary buyer and wedge.
- Create sales deck and one-page PDF.
- Build referral and partner motion.
- Convert assessment patterns into content series.
- Identify one publishable case study with permission or safe anonymization.

### Validation Questions

- Is Human API the category name buyers repeat back?
- Which offer converts fastest?
- Which findings create urgency?
- Which remediation work is most common?

## 12-Month Plan

### Advisory Business

- Establish Human API Risk Assessment as a repeatable TKO diagnostic.
- Build a pipeline from assessment to decision-layer sprint to fractional advisory.
- Develop industry-specific variants for prior authorization, UM, care management, and transformation offices.
- Explore Enterprise Value & Exit Readiness as a future assessment variant for founders, healthcare practice owners, PE, search funds, business brokers, M&A advisors, and fractional operators.
- Create a benchmark-informed assessment model if enough client data exists.

### Product Exploration

Only consider software exploration if:

- Multiple assessments use the same artifacts and scoring patterns.
- Clients ask for ongoing tracking.
- Manual delivery reveals repeatable data structures.
- A clear budget exists for monitoring or reassessment.

Potential software should start as internal tooling, not a client-facing platform.

### Prior Authorization Assessment Prototype

Future opportunity only. No application, schema, integration, or AI implementation should start before the assessment has manual proof.

Phase 1:

- Spreadsheet plus lightweight web application.
- Conceptual entities: Provider, Payer, Authorization, Denial Reason, Exception, Staff Member.
- Capabilities: track denials, categorize denials, identify repeat patterns, identify Human APIs.
- Explicit exclusions: no AI, no FHIR, no claims integration.

Phase 2:

- Add workflow mapping.
- Add bottleneck analysis.
- Add administrative burden scoring.

Phase 3:

- Add Gold Card Readiness Score.
- Example dimensions: Documentation Quality, Approval Rate, Exception Dependency, Workflow Standardization.

### Long-Term Strategic Option

Human API Risk Assessment could become the front door to a broader Operational Intelligence platform, but only if the advisory work proves:

- The problem is urgent.
- Buyers pay for diagnosis.
- Remediation patterns repeat.
- Data can be captured consistently.
- Ongoing monitoring has budget and operational value.

Enterprise Value & Exit Readiness should remain a future advisory path until the diagnostic method has proof. It should document founder dependency, key-employee dependency, operational memory, workflow scalability, and AI readiness as value risks; it should not become generic valuation, broker, or M&A advisory work.
