# Operational Recovery Assessment — One-Pager (Requirements)

**Purpose:** Define the content requirements for the Operational Recovery Assessment
one-pager — the single conversion asset that will later be produced as a **PDF**, a
**landing page**, and a **sales-deck asset**. This is a **requirements specification**, not
the final designed asset.

**Alignment:** Offer 0 in [`CURRENT_REALITY.md`](CURRENT_REALITY.md) (1 week, $5K–$8K).
Backed by the [Recovery Assessment Playbook](content/offers/recovery-assessment/RECOVERY_ASSESSMENT_PLAYBOOK.md)
and the [Assessment Conversion System](ASSESSMENT_CONVERSION_SYSTEM.md).

**Honesty constraints:** No fabricated metrics, no client names, no unverifiable claims.
Proof is RachelOS (deployed, code-backed) and honest advisory framing for healthcare.

**Downstream formats this must serve:**
- **PDF** — leave-behind / email attachment after discovery calls
- **Landing page** — the public conversion page (see Conversion System §Landing Page)
- **Sales deck asset** — 1–2 slides condensed from this content

---

## Required Content Blocks

The one-pager **must include** these six blocks, in this order:

### 1. Problem Statement
- Lead with the "human API" failure mode in the buyer's language.
- Core line (approved framing): *"Your operation has data, tools, and process — but critical work still runs through a few key people. When they're overloaded or out, execution stalls."*
- Name the symptoms: knowledge in heads not systems; "who needs attention?" can't be answered without a meeting; status looks green while risk builds; AI pilots that demo well and change nothing.

### 2. Who It Is For
- Target titles: COO / VP Operations; VP Revenue / Customer Success; PE operating partner; owner-operator of a relationship-driven business; healthcare operations leader (warm/referral, advisory framing).
- One honest disqualifier line so it self-selects: *not* for buyers who want commodity automation, a model-selection project, or a platform.

### 3. Deliverables
The six Offer 0 deliverables, verbatim in intent with `CURRENT_REALITY.md`:
1. Workflow map
2. Bottleneck analysis
3. Dependency analysis (where the operation depends on specific people)
4. AI readiness assessment
5. Operational risk assessment
6. Executive briefing

### 4. Timeline
- **1 week**, fixed scope. Optionally show the 5-day shape (scope → interviews → analysis → synthesis → readout) from the Playbook.
- Set price expectation: **$5K–$8K**.

### 5. Outcomes
- What the buyer walks away with — stated as operating conditions, **not metrics**:
  - A clear, evidence-based view of the top 3 places work stalls.
  - Identification of the key-person dependency and the risk it creates.
  - An honest read on where AI can safely help (human-approved) and where it can't yet.
  - The single highest-leverage next move.
- **No invented numbers.** A `[VERIFIED OUTCOME — TBD]` slot may be added once a real,
  attributable result exists.

### 6. Next Step
- Single, clear CTA: *Book a 30-minute discovery call* (not the assessment directly — per the Conversion System scheduling flow).
- Reassurance line: fixed scope, fixed week, fixed price; clear path afterward (Assessment → Diagnostic → Build → Advisory) with no obligation to continue.

## Required Proof Element (must appear)

- A **RachelOS proof strip**: one line + one visual placeholder establishing that TKO has
  *already built* a deployed Operational Knowledge System — "Deployed, runs daily,
  human-approved." Visual sourced from the
  [Visual Asset Master Plan](VISUAL_ASSET_MASTER_PLAN.md) (Queue / Digest / Needs-Rachel).
- This is what separates the one-pager from a generic consulting flyer: the offer is backed
  by a working system, not just a methodology.

---

## Tone & Constraints

- Plain, operator-to-operator; skeptical-of-hype; concrete.
- No metrics that aren't verifiable; no client names; no overstated AI claims.
- Healthcare references use advisory/experience framing only.
- Framework spine stage names, if shown, must match `CURRENT_REALITY.md` exactly.

## Format-Specific Notes (for production later)

| Format | Constraint |
|---|---|
| **PDF** | Fits on one page (or one front/back); print-safe; RachelOS visual embedded; CTA = discovery-call link/QR. |
| **Landing page** | Expands blocks into the Conversion System landing-page sections; CTA = intake form / scheduling. |
| **Sales deck** | Condense to 1–2 slides: Problem + Human-API on one, Deliverables/Timeline/Outcomes/Next-Step on the other; RachelOS proof visible. |

**Definition of done (requirements stage):** every block above has approved copy direction
and an identified proof/visual input, so a designer/copywriter can produce all three formats
without further strategic discovery.
